# Content Standards

มาตรฐานนี้ใช้ควบคุมโทนและคุณภาพของหนังสือ

## 1. ประเภทของข้อมูลในหนังสือ

ทุกข้อมูลควรถูกเขียนให้อยู่ในกลุ่มใดกลุ่มหนึ่ง:

- `Official behavior`
  พฤติกรรมของ Revit หรือ Starlight ที่ควรอ้างแบบทั่วไป
- `Project example`
  ค่าและโจทย์จากงานตัวอย่าง
- `Engineering judgment`
  คำอธิบายเชิงหลักการที่ต้องระวังไม่พูดเกินจริง

## 2. ภาษาที่ควรใช้

- ใช้ภาษาไทยที่อ่านง่าย
- อังกฤษใช้เฉพาะชื่อคำสั่ง เมนู และคำเทคนิคที่จำเป็น
- หลีกเลี่ยงถ้อยคำที่ฟันธงเกินไป ถ้าข้อมูลขึ้นกับเวอร์ชันหรือบริบท

## 3. รูปแบบบทที่ดี

บทที่สอนใช้งานควรมีโครงแบบนี้:

1. บอกเป้าหมายของบท
2. บอก prerequisite ถ้ามี
3. ให้ขั้นตอนทีละลำดับ
4. มี checkpoint หรือผลลัพธ์ที่ควรเห็น
5. มี warning จุดผิดบ่อย
6. ปิดด้วยการเชื่อมไปบทถัดไป

## 4. สิ่งที่ควรหลีกเลี่ยง

- ค่าในตารางที่ขัดกันเอง
- ชื่อ level/grid/type ไม่สอดคล้องกัน
- คำอธิบายทางวิศวกรรมที่ทำให้เข้าใจหน้าที่ของเหล็กผิด
- สอนชื่อ node, plugin หรือเมนูแบบตายตัวโดยไม่บอกข้อจำกัดเรื่องเวอร์ชัน

## 5. การใช้รูปภาพ

- ถ้ามีการอ้างรูป ต้องมีไฟล์จริงใน `public/images/...`
- ถ้ารูปหาย ให้ลบ reference หรือแทนด้วยคำอธิบายชั่วคราว
- รูปที่ดีควรช่วยให้คนอ่านตรวจ checkpoint ได้

## 6. การแก้บท advanced

บทระดับกลางถึงสูง เช่น Family, Dynamo, Coordination ควรมี:

- prerequisite box
- ข้อจำกัดของ workflow
- บอกชัดว่าอะไรเป็นแนวคิด อะไรเป็นขั้นตอนที่ทำตามได้ตรงๆ

## 7. มาตรฐานก่อน merge หรือปิดรอบแก้

- build ผ่าน
- ไม่มี warning ใหม่จากงานรอบนั้น
- เนื้อหาที่แก้ไม่ชนกับบทอื่น
- ถ้าปรับหลักการหรือค่า example ต้องตรวจบทที่อ้างค่าชุดเดียวกันด้วย
