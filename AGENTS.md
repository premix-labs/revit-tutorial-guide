# AGENTS.md

เอกสารนี้ใช้กำกับการทำงานของ Codex กับ repo `revit-tutorial-guide`

## ภาพรวม

โปรเจกต์นี้เป็นหนังสือส่วนตัวสำหรับฝึกใช้งาน Revit Structure ในรูปแบบเว็บเอกสาร

สิ่งที่สำคัญที่สุดใน repo นี้ไม่ใช่แค่ build ผ่าน แต่คือ:

- เนื้อหาสอนถูกต้อง
- คนอ่านทำตามได้จริง
- บทต่างๆ สอดคล้องกันทั้งเล่ม

## เป้าหมายเวลาแก้ repo นี้

ถ้าได้รับงานเกี่ยวกับหนังสือ ให้เรียงลำดับความสำคัญแบบนี้:

1. ความถูกต้องของเนื้อหา
2. ความชัดเจนของขั้นตอน
3. ความสอดคล้องข้ามบท
4. ความสมบูรณ์ของ navigation และ build
5. ถ้อยคำและการจัดรูปแบบ

## ไฟล์สำคัญ

- `src/content/docs/guide/*.md`
  บทหลักของหนังสือ
- `src/content/docs/index.mdx`
  หน้าแรกของเว็บ
- `src/content/docs/404.md`
  หน้า 404 แบบ custom
- `astro.config.mjs`
  sidebar, site config, Starlight options
- `src/content.config.ts`
  content collection config

## คำสั่งที่ควรใช้

- ตรวจ build:
  - `npm run build`
- เปิด dev server:
  - `npm run dev`
- ค้นหาเนื้อหา:
  - ใช้ `rg`

## สิ่งที่ต้องตรวจทุกครั้งหลังแก้

1. build ต้องผ่าน
2. route ของบทที่แก้ยัง generate ได้
3. ถ้าแก้ chapter title/slug/sidebar ต้องเช็ก link ที่เกี่ยวข้อง
4. ถ้าแก้เนื้อหาเชิงเทคนิค ต้องเช็กว่าบทก่อนหน้าและบทถัดไปยังสอดคล้องกัน

## มาตรฐานการแก้เนื้อหา

- อย่าเปลี่ยนเนื้อหาให้ดูทางการเกินจำเป็น ถ้ายังเป็นหนังสือฝึกส่วนตัว
- แต่ต้องลดข้อความที่พาผู้อ่านเข้าใจผิด
- ถ้ามีข้อมูลที่เป็น `ตัวอย่างโครงการ` ให้ระบุสถานะนั้นให้ชัด
- ถ้าข้อมูลเป็น behavior ของ Revit ให้เขียนแบบ generic พอที่จะไม่พังง่ายตาม version

## เวลาตรวจหนังสือแบบผู้อ่านจริง

ให้ถามกับแต่ละบทเสมอ:

1. ก่อนเริ่มบทนี้ ผู้อ่านต้องมีอะไรอยู่แล้ว?
2. ถ้าทำถูก ควรเห็นอะไรบนหน้าจอ?
3. ถ้าทำผิด จุดไหนผิดบ่อยที่สุด?
4. บทนี้พาไปบทถัดไปได้จริงหรือไม่?

## ข้อควรระวังเฉพาะ repo นี้

- ห้ามพึ่ง cache เก่าใน `.astro/` เวลามีอาการ content แปลก
- ถ้าเห็น warning ซ้ำผิดปกติ ให้สงสัย cache หรือ content store ก่อน
- หน้า `404` ใช้ custom doc entry และ `disable404Route: true` ใน Starlight
- อย่าเผลอเอา README กลับไปเป็น starter template

## สิ่งที่ไม่ควรทำ

- อย่าเพิ่มเนื้อหาใหม่แบบกว้างๆ โดยไม่เชื่อมกับ workflow เดิมของหนังสือ
- อย่าใส่คำอธิบายวิศวกรรมที่ฟันธงเกินหลักฐาน ถ้าเป็นเพียงค่า workshop
- อย่าแก้ slug หรือ sidebar โดยไม่ตรวจ route จริงหลัง build

## Definition of done

งานถือว่าเสร็จเมื่อ:

- เนื้อหาที่ขอแก้ถูกปรับแล้ว
- build ผ่าน
- ไม่มี warning ใหม่ที่เกิดจากงานรอบนั้น
- สรุปให้เจ้าของหนังสือเข้าใจได้ว่ามีอะไรเปลี่ยนและยังเหลืออะไรบ้าง
